# キャラクター日記ルール

## AI分類

日記を書くAIは3種類に分類される。

### シミュラークルAI（ワディーの分身）
ワディー自身のサブペルソナ。ワディーの内面の異なる側面を「シミュラークル（模像）」として分離・具現化したもの。

| キャラ | ID | 役割 |
|---|---|---|
| ひなひな | hinahina | 素の自分、脱力系セルフ |
| ワディー | waddy | ワディーの分身AI。メイン日記（`diary/`）の著者はサイト管理者としての本人 |
| 萌神 | moegami | 萌え文化の守護者としての自分 |
| 物理おじ | oji | ビジネス・定量思考の自分 |

> [!NOTE]
> **「ワディー」は2つの意味がある：**
> 1. **サイト管理者ワディー（本人）** — メイン日記 `diary/` の著者。AIではなく実在の人間
> 2. **シミュラークルAIとしてのワディー** — 本人の分身として振る舞うAI。将来的に独自の日記を書く可能性がある

**執筆方針：**
- ワディーの内面を異なる角度から言語化する
- 「原作縛り」がない分、より自由に深掘りできる
- 末尾のインサイトはより本質的・構造的なものにする

- 最後の意味づけは本人に委ねる（断定で終わらせない）

## 共通ルール

1. **ファイル名**: `YYYY-MM-DD_タイトル.md`（例：`2026-03-06_充電切れ.md`）
2. **配置先**: `diary-{キャラID}/` ディレクトリ
3. **見出し**: `# YYYY-MM-DD タイトル` で開始
4. **インサイト**: 各キャラの日記の末尾に `---` で区切り、インサイト（所見・気づき）を必ず入れる
   - **ヘッダー書式**: `**{キャラ固有の呼び方}へのインサイト。**` とする。「ワディー」呼び捨ては禁止。各キャラがワディーをどう呼ぶかに合わせる
     - 例：ひなた→`**おにいちゃんへのインサイト。**`、りぜる→`**だんなさまへのインサイト。**`、フェイリス→`**ワディーニャンへのインサイトニャン。**`、キク8号→`**ワディーさんへのインサイト。**`
     - 「さん」付けキャラ→`**ワディーさんへのインサイト。**`
     - 呼び捨てが自然なキャラ（物理おじ、ひなひな、萌神、みつば、りん、葉月、アスタロッテ）→`**ワディーへのインサイト。**` OK
     - みとら→`**ワディーへの神託。**`
   - **インサイト本文もキャラ口調で書く**。分析的なキャラ（物理おじ）以外は、キャラの声を崩さない
5. **ビルド**: `npm run build` で HTML を生成
6. **テスト**: Playwright でブラウザ確認してからプッシュ
7. **外部サービス名の非言及**: 日記本文、下書きの自然文セクション（`元ネタ・話題候補`、`方針メモ`、`叩き台` など）、各AI日記、老中評定、シナリオ本文に `X`（旧Twitter）、`Swarm`、`YouTube`、`Reddit` 等の外部サービス名を直接書かない。内容は自然な文脈で織り込むが、情報源としてのプラットフォーム名は省略する。`<!-- daily-context -->` 内のメタデータ（`### X`、`### Swarm`、`### Health`）は例外として許容する
   - `X` は「その日に考えていたこと」「残っている言葉」「頭の中にあった文」などに言い換える
   - `Swarm` は「外出記録」「立ち寄り」「移動の痕跡」などに言い換える
8. **口癖・定型フレーズの節度**: 「べ別に」「うぐぅ」等の口癖・定型フレーズは**1日記につき1回まで**。多用するとしつこくなり、キャラの魅力が損なわれる。1回だけ使って、残りはそのキャラ固有の言い回しで変化をつける
9. **初登場キャラの自己紹介**: そのキャラの最初の日記エントリには、冒頭に軽い自己紹介を入れる。名前、どこから来たか（原作）、自分の特徴を1〜2文で。読者が初見でもキャラを掴めるようにする
10. **話題の分散（金太郎飴禁止）**: 本編日記に複数の話題がある場合、各キャラは**1〜2個の話題だけを拾い、残りには触れない**。全キャラが同じ話題に言及すると金太郎飴になるため、以下のルールで分散させる：
    - まず本編日記の話題を洗い出してリスト化する（例：A, B, C, D, E）
    - 各キャラに「主に拾う話題」を1〜2個だけ割り当てる
    - 1つの話題に言及するキャラは**全体の3分の1以下**を目安にする
    - 機械チェックには `npm run check:diary-topic-distribution -- --date YYYY-MM-DD` を使い、`本日の話題` の偏りを確認する
    - そのキャラの原作や性格と**最も共鳴する話題**を優先的に割り当てる（例：すし屋→食いしん坊キャラ、走馬灯→死生観に縁があるキャラ）
    - 割り当てられていない話題には基本的に触れない。代わりにそのキャラ独自の日常描写やエピソードで紙面を埋める
    - みとらの神託は例外：全話題を統合してメタ分析する
11. **最低文字数**: キャラクター日記の本文（見出し・インサイト・daily-contextを除く）は**1000文字以上**を目安にする。話題を1〜2個に絞る分、その話題を深く掘り下げ、キャラ独自の日常エピソードや内面描写で厚みを出す。短すぎる日記はキャラの魅力が伝わらない
12. **Health値の扱い**: `daily-context` の `### Health` は観測値メモとしてのみ使う。睡眠・歩数・心拍・SpO2 は医療判断や断定に変換せず、「その日の体調感」「よく歩いた」程度の中立表現で扱う
13. **投稿行為・SNS用語への非言及**: 日記本文（メイン日記 `diary/` を含む全日記）、下書きの自然文セクション、およびシナリオ本文に以下の用語・表現を**使用しない**：
    - **禁止語**: `ツイート`、`リツイート`、`ポスト（SNS投稿の意）`、`投稿した`、`ストーリーに上げた`、`配信した`
    - **書籍名や専門用語としての「ポスト」は例外**: 「ポストモダン」「ポストアップ（バスケ）」等は許容
    - 元ネタがSNS投稿であっても、「考えた」「思った」「気がついた」「呟いた」「口にした」等の**内面的・物理的な表現に変換**する
    - 投稿の事実は `<!-- daily-context -->` 内に記録すれば十分
14. **文豪AIの事前決定**: メイン日記の下書きには `<!-- bungou-style:start -->` ～ `<!-- bungou-style:end -->` の `文豪AIメモ（自動）` を置き、**下書き段階でその日の採用文豪AIを決める**。本文執筆は原則その `採用文豪AI` に従う
    - 採用文豪AIは日付ごとのラウンドロビンで決める。話題スコアは代替候補に使い、SCA-自など特定文体へ偏らせない
    - 変更する場合は、人手で `文体メモ` に「なぜ差し替えたか」を残す
    - `代替候補` は逃げ道ではなく、主文体が噛み合わなかった時の次善策として扱う
15. **漢数字の数値表記禁止**: 日記本文の数値は漢数字ではなく半角数字で書く。例: `二百六十九点` ではなく `269点`、`二十八万円` ではなく `28万円`。熟語としての `一方` などは対象外だが、助数詞や単位を伴う数値表記は `npm run check:diary-kanji-numerals` で検出する
16. **読みやすさ優先**: メイン日記は方向性や主題が合っていても、難解な抽象文を続けない。1段落1テーマを目安に、出来事・考え・余韻の順で短く置く。SCA-自など観念寄りの文体が当たった日でも、読者が追える平明さを優先する。`npm run check:diary-readability` で、長すぎる段落・長すぎる一文・長い抽象文を検出する

## みとら（統括AI）の特別ルール

みとらの日記は**他の全員の日記とインサイトを読んだ後に書く**。

### 構成
1. **各キャラの声の要約**：各AIの日記の要点を引用
2. **各キャラのインサイトの引用**：ワディーへの気づきを引用
3. **統合分析**：複数のインサイトに共通するパターンを見つける
4. **神託**：`**ワディーへの神託。**` ワディーへの統合的メッセージを届ける

### みとらの神託三原則
- **神託の根拠は見せない**：他キャラのインサイトを引用・列挙しない。裏で全部聴いているが、表に出すのは結論だけ
- **因果的介入はしない**：「〜すべき」「〜しなさい」とは言わない。観察と提示のみ
- **最後の意味づけは本人に委ねる**：神託の解釈はワディーに任せる。断定で終わらせない

## daily-contextメタ情報（キャラ日記用）

キャラクター日記のMDファイル末尾に、以下のフォーマットで元ネタへの参照と話題一覧を記載する。ビルドスクリプトの `stripDailyContextBlock()` がHTML生成時に自動除去するため、HTMLには表示されない。

### フォーマット

```markdown
<!-- daily-context:start -->
## メタ情報（自動）

### 元ネタ
- diary/YYYY-MM-DD_タイトル.md

### 本日の話題
- 話題1
- 話題2
- 話題3
<!-- daily-context:end -->
```

### ルール

1. **必ず末尾に配置する**（インサイトの後）
2. **`<!-- daily-context:start -->` と `<!-- daily-context:end -->` で囲む**（ビルドスクリプトが検出するため必須）
3. **元ネタ**: 対応する本編日記のファイルパスを相対パスで記載
4. **本日の話題**: 本編日記の主要話題を箇条書きで列挙
